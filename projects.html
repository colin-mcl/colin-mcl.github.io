Universal Machine

The Universal Machine was a program that we created in COMP 40, one of the most challenging computer science courses offered at Tufts. The task was to create a program that emulated the architecture and implementation of a low-level computer. Our program would be fed input files in raw binary which could be converted to 32-bit instructions. These instructions emulated assembly language and would perform basic operations on a set of memory registers. Some operations included waiting for input, printing, basic arithmetic and halting the program. The most difficult requirement was the ability to map new memory segments in which future instructions could be stored. With this implemented, complicated programs could be run on our emulation by creating segments which stored entire programs, then loading that segment into the instruction register.

Implementing the Universal Machine helped us gain a much better understanding of how the execution of a computer program actually occurs. The managing of registers and memory created many challenges but with careful design and rigorous unit testing we created a great emulation. After the initial project was finished we analyzed the performance of the UM using Callgrind and KCachegrind. We were able to eliminate many code bottlenecks and create an 88% speedup in our program. Finally, we used a macro version of the UM assembly language to create a working RPN calculator.

    Skills Involved:
  • C
  • Memory Management
  • Bit Manipulation
  • Modular Design

Sudoku

This is a command line C++ program which takes a Sudoku board formatted in text as input. The board can have any number of squares filled in and the program will solve the remainder of the board using a backtracking algorithm. The algorithm will recursively fill in every square on the board. It will attempt every solution possible at every square and check if this leads to a solved board. If it does lead to a solution the algorithm continues. Otherwise, it removes that attempt, tries the next possibility and so on. Instead of using an array of integers to store the numbers that are present in each row, column and square, the program represents each row, column and square with a bitpacked integer. This means that the least-significant 9 bits of each integer represent whether numbers 1-9 have been found yet. This reduces the memory required to store the numbers which have been found on a sudoku board from about 7000 bits to just 864. While this does not have a significant impact on the program's runtime, our lessons in locality and performance from COMP 40 incentived me to use this solution.
